FlexNeRFer is a general-purpose NeRF accelerator for real on-device rendering budgets.

The paper starts from a simple systems problem: existing NeRF accelerators tend to overfit one rendering pipeline, while GPUs miss the area, power, and latency targets of portable devices. FlexNeRFer answers that with a hierarchical multi-dataflow NoC, a precision-scalable MAC array, and runtime selection of the right sparsity format for each workload tile.

Abstract

The paper is not pitching one faster NeRF. It is pitching hardware that survives the diversity of modern NeRFs.

NeRF has moved beyond one canonical model into a family of renderers with different network structures, encodings, sparsity patterns, and precision needs. That is the paper's real starting point: specialized accelerators can be excellent for one NeRF variant and then lose most of that advantage when the next model changes the computation mix.

FlexNeRFer is designed as a broader accelerator substrate. It profiles seven representative NeRF models, identifies GEMM/GEMV and encoding as the dominant bottlenecks, then builds an accelerator around those common kernels. The key hardware ideas are a hierarchical mesh-with-feedback NoC that supports unicast, multicast, and broadcast on a precision-scalable MAC array, plus an adaptive compression path that changes data format based on precision mode and sparsity ratio instead of hard-coding one sparse layout.

Motivation

Three constraints shape the entire design: model diversity, sparse execution, and portable-device budgets.

The page should make clear that FlexNeRFer is a response to a moving target. NeRF workloads differ enough that the architecture has to stay flexible without losing efficiency.

NeRF models do not share one fixed computational pattern

Across the profiled models, the architecture has to handle MLP-heavy pipelines, CNN-style feature extraction, transformer-style attention, positional or hash encoding, and model-specific sparsity tricks. A single rigid accelerator leaves performance on the table as soon as the model family shifts.

The sparse format that saves memory is not constant

The paper shows that COO, CSC/CSR, bitmap, and uncompressed layouts win under different sparsity ratios and precision modes. That means a useful accelerator has to choose formats adaptively instead of assuming one sparse representation is always best.

Portable devices need hard area and power limits

The target is not a datacenter GPU. The paper explicitly frames on-device hardware around roughly sub-100 mm2 area and sub-10 W power, because AR glasses, VR systems, and similar edge devices cannot absorb desktop-class rendering hardware.

Pipeline

FlexNeRFer is a systems stack: profile the bottlenecks, then build flexibility into the datapath, the reduction path, and the memory format.

The method is not one clever block. The architecture is useful because several pieces are aligned to the same observation: NeRF workloads are irregular, sparse, and precision-variable.

Step 01
Profile many NeRFs and isolate the shared bottlenecks
The paper measures seven representative NeRF models on an RTX 2080 Ti and finds that GEMM/GEMV dominates runtime across the board, while encoding also becomes a major cost for models such as KiloNeRF, NSVF, Mip-NeRF, and Instant-NGP. That profiling step is what justifies a general accelerator rather than another model-specific design.
Step 02
Use HMF-NoC to feed sparse data into a bit-scalable MAC array
At the core of the GEMM/GEMV unit is a hierarchical mesh with feedback NoC. It supports 1D unicast, multicast, and broadcast so sparse and irregular tensors can still be mapped densely across the array. The same MAC substrate supports 4-bit, 8-bit, and 16-bit modes instead of locking the design to one numerical format.
Step 03
Adapt the sparsity format to the data tile and precision mode
FlexNeRFer measures input sparsity ratio online and pre-analyzes weight sparsity offline, then picks the compression format with the lowest memory footprint for that operating point. This is one of the paper's most specific insights: the best format shifts as precision and sparsity change, so adaptive compression directly improves both bandwidth and energy.
Step 04
Accelerate both encoding styles used in modern NeRFs
The architecture includes a positional encoding engine and a hash encoding engine. The positional engine uses arithmetic approximations and processes 64 encodings in parallel, while the hash engine extends NeuRex-style lookup with 64 coalescing hash units, 64 subgrid hash units, and 64 interpolation units to reduce redundant memory traffic.

Why HMF-NoC matters

The paper compares the proposed mesh-with-feedback NoC to HM-NoC and reports about 2.5x lower on-chip memory-access energy. The important point is not just connectivity; it is that feedback and hierarchical distribution reduce the cost of keeping the MAC array usefully busy under sparse mappings.

Why precision and sparsity are coupled

FlexNeRFer is not merely a low-bit accelerator with optional sparse support. The work explicitly studies how memory footprint shifts when the same tensor moves from 16-bit to 8-bit to 4-bit, and uses that to decide when COO, CSC/CSR, bitmap, or no compression is preferable.

Results

The strongest result is that the accelerator stays within edge-style area and power while still beating both GPUs and a recent NeRF accelerator by wide margins.

This is a layout-backed hardware paper rather than a pure architecture sketch. The reported numbers come from 28nm implementation, post-layout modeling, and comparisons across seven NeRF models on two standard datasets.

The area overhead is deliberate, not accidental

Compared with NeuRex, FlexNeRFer spends more silicon and power on flexibility: the paper reports 48.4 percent area overhead and 35.18 percent power overhead. But the compute-density result is still better, at roughly 1.9x to 7.5x higher performance per area unit, because that flexibility keeps the array useful across more workloads.

The encoding engines matter more than a generic summary suggests

The positional encoding engine alone is reported to achieve 8.2x area reduction and 12.8x power reduction versus a DesignWare IP-based implementation. That matters because encoding is a major fraction of runtime in several NeRF variants, not a minor auxiliary step.

The paper treats precision as a quality-efficiency curve, not a fixed choice

The sensitivity study shows INT16 stays close to FP32 PSNR, while low-bit quantization needs outlier-aware rescue to recover quality. That is a useful systems message: the accelerator is not only fast at low precision, it gives multiple operating points that preserve image quality differently.

Takeaways

Why this project matters beyond one rendering benchmark.

  1. Point 01
    This is a hardware paper about model diversity. The core contribution is not just a faster NeRF engine, but an accelerator architecture that remains efficient as NeRF variants change their encoding, sparsity, and precision behavior.
  2. Point 02
    The page should emphasize that FlexNeRFer is genuinely on-device by design. The reported area and power are part of the argument, not secondary implementation details.
  3. Point 03
    For CELL's website, this project is a clear example of deployment-aware accelerator design: profile the real workloads, identify what changes across models, and make the architecture flexible exactly where those changes occur.
