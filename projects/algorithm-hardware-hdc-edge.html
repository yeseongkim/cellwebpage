Abstract

Making HDC run on hardware people can actually buy.

Hyperdimensional computing is attractive for edge learning because training and inference are much lighter than deep neural networks, but the common hardware story used to depend on custom accelerators that are not easy to deploy in the field. This paper takes the opposite route: instead of designing new silicon first, it adapts HDC to the matrix-heavy primitives already available on low-power edge ML hardware.

The key idea is to reinterpret HDC as a hyper-wide neural network. Encoding and similarity-heavy inference are mapped to TensorFlow and Edge TPU execution, while the remaining class-hypervector updates stay on the host CPU. On top of that, bagging reduces CPU-side training work without sacrificing final quality. The result is a practical edge-learning framework that is 4.5x faster for training and 4.2x faster for inference than the baseline platform, and 19.4x and 8.9x faster than a similar-power Raspberry Pi baseline.

Motivation

Why this paper mattered for edge HDC.

The paper is less about inventing a new HDC rule and more about showing that HDC can be shaped to fit mainstream low-power ML hardware rather than waiting for a custom accelerator pipeline.

Custom accelerators were the bottleneck

Earlier HDC acceleration stories often relied on ASIC-style hardware proposals. Those designs can be compelling academically, but they are slow to fabricate and difficult to deploy when an application changes.

Edge TPU is useful but incomplete

Google Edge TPU already provides efficient low-power inference, but it does not support the full training flow needed by HDC, especially class-hypervector updates. The framework therefore has to split work intelligently between TPU and CPU.

Wide hypervectors still cost time

Even though HDC is lightweight compared with DNNs, encoding with 10,000-dimensional hypervectors still involves large matrix operations. Those computations are exactly where commodity accelerators can help if the algorithm is mapped correctly.

Pipeline

How the framework splits HDC across CPU and Edge TPU.

The system view is simple: put the dense, accelerator-friendly parts onto the TPU, leave the unsupported incremental updates on CPU, and then reduce the CPU burden with an ensemble trick that still collapses back into one inference model.

Step 01
Interpret HDC as a hyper-wide neural network
The paper maps the HDC encoder to the first half of a three-layer fully connected network. Input features become the input layer, the 10,000-dimensional hypervector space becomes a wide hidden layer, and the randomly generated base hypervectors become the weight matrix.
Step 02
Push encoding and inference onto Edge TPU
Because HDC encoding is fundamentally a large vector-matrix multiply plus a nonlinearity, it fits TensorFlow Lite execution well. The same mapping idea also turns the final similarity-to-class computation into an Edge TPU-friendly matrix operation for inference.
Step 03
Keep class-memory updates on CPU
Edge TPU does not support on-device weight updates or the element-wise operations needed for class hypervector training. The framework therefore keeps bundling and detaching updates on the host CPU, where the class memories are adjusted from encoded samples.
Step 04
Use bagging to shrink CPU-side work
Bagging creates several weaker submodels trained on bootstrap-sampled subsets. That allows fewer training iterations and reduced per-model cost, then merges the learned base and class hypervectors back into a single full inference model with no extra inference-time ensemble overhead.
Results

The practical win is that edge learning becomes fast without losing HDC's accuracy profile.

The experiments cover five datasets and compare three setups: CPU-only baseline, TPU acceleration without bagging, and the full TPU-plus-bagging framework. The strongest gains appear when the feature dimension is large enough for the Edge TPU to stay busy.

Dataset-level runtime Training speedups include 3.49x on FACE, 2.45x on ISOLET, 1.81x on UCIHAR, and 4.49x on MNIST; inference speedups include 3.16x, 2.13x, 3.08x, and 4.19x on those same datasets. The full framework helps both phases, but especially when input vectors are wide enough to benefit from TPU matrix throughput.
Accuracy retention The Edge TPU versions retain similar inference accuracy to the CPU baseline, and the bagging-based version can even improve over the full model on datasets such as ISOLET and PAMAP2. The speedup is not purchased by accepting a worse final classifier.
Bagging parameters The paper settles on dataset sampling ratio alpha = 0.6, disables feature sampling, and uses 6 training iterations for the bagging setting. The ensemble is used as a runtime optimization, not as a heavyweight accuracy-maximizing ensemble at inference time.
Failure mode PAMAP2 shows weaker gains because it has far fewer input features than the other datasets. This makes clear that Edge TPU acceleration is most valuable when encoding is large enough to amortize transfer and setup overhead.

An early deployment-first HDC page

The paper's importance is that it reframes HDC as something that can live on standardized ML stacks. It does not wait for a custom accelerator ecosystem to appear before trying to solve edge deployment.

Why the co-design matters

This is not a simple "run it on TPU" story. The authors are explicit that the unsupported training path remains on CPU, then engineer the algorithm around that boundary with bagging so the system still works end to end.

Takeaways

Why this page matters in CELL's early HD systems arc.

  1. Point 01
    This is one of the clearest early project pages for CELL's "efficient learning on existing hardware" philosophy. The contribution is not only an HDC method but a deployment recipe.
  2. Point 02
    The page should emphasize co-design more than raw acceleration. Mapping encoding and inference to Edge TPU while reshaping training around the CPU boundary is the actual systems insight.
  3. Point 03
    Chronologically, it is also a strong anchor for the lab's later work on trainable, hardware-aware, and edge-oriented hyperdimensional learning. The Best Paper Award note is worth keeping visible.
