Abstract

The paper is solving three robotics problems at once: edge efficiency, federated privacy, and imbalanced action data.

Federated learning is attractive for mobile robots because their data are naturally distributed and often privacy-sensitive, but conventional federated pipelines still assume deep neural models that are expensive for edge platforms. Hyperdimensional computing offers a more lightweight alternative, yet prior HD federated methods typically keep their encoders static, which hurts both representation quality and privacy.

This paper introduces a federated HDC framework built for LiDAR-equipped wheeled robots. The first key idea is dynamic encoding through HSL, a stochastic update rule that continuously adjusts the projection matrices without resorting to full backpropagation. The second is Hypervector Oversampling, which synthesizes minority examples directly in hyperspace rather than on raw sensor inputs. Together they let four mobile robots collaborate on object avoidance, object tracking, and terrain navigation while keeping the model efficient enough for embedded deployment.

Motivation

Static encoders are the bottleneck the paper is pushing against.

The work is not simply "federated learning with HDC." It is specifically about why static encoders and raw-data oversampling are a bad fit for real robotic federated systems.

Deep FL is heavy for small robots

Mobile robots have limited compute and energy budgets. The paper positions HDC as a lightweight alternative to DNN-based federated learning, with prior work showing orders-of-magnitude efficiency advantages in constrained environments.

Static encoders leak too much and adapt too little

Fixed projection matrices can be exploited for model inversion or data reconstruction, and they also produce suboptimal representations when local data distributions shift. Dynamic encoding is therefore framed as both an accuracy improvement and a privacy improvement.

Robot behavior data are naturally imbalanced

Tasks such as obstacle avoidance and tracking generate many more stop actions than rare directional responses. Without balancing, the model overfits the majority class and under-represents the actions that matter in more difficult situations.

Pipeline

The system has two real technical moves: HSL for encoding and HVO for balancing.

The page should be explicit about what changes compared with ordinary HD federated learning. The method is defined by how local encoders are updated and how minority classes are synthesized directly in hyperspace.

Step 01
Each robot encodes LiDAR observations into hypervectors
The four wheeled robots use RPLiDAR A1 sensors with 720 points per scan at 0.5-degree resolution. Sensor-action demonstrations are encoded into 6000-dimensional hypervectors and stored locally for the object avoidance, object tracking, and terrain navigation tasks.
Step 02
HSL updates the encoder without backpropagation
Instead of relying on TrainableHD-style backpropagation, HSL introduces a random hypermatrix B and updates the encoder stochastically based on similarity scores. The paper writes this as E ← E ⊕ (s × B × λ), which lets the projection matrix adapt over time with lower cost and without requiring raw-input gradients.
Step 03
HVO synthesizes minority samples in hyperspace
To handle class imbalance, the method does not use SMOTE on raw data by default. It instead creates synthetic hypervectors directly from minority-class encodings, ultimately using a simple half-and-half concatenation of two existing hypervectors because it performs similarly to richer interpolation while being cheaper to compute.
Step 04
Clients train locally, then the server averages encoders and class hypervectors
After local retraining, each client uploads its encoder and class hypervectors. The server averages them into a global model and broadcasts the result back to clients for the next global round, preserving the usual federated-learning structure without exchanging raw LiDAR data.

Why HVO matters in HDC specifically

Encoding is usually the most expensive part of HD learning. Because HVO works directly on already-encoded hypervectors, it avoids generating more raw samples that would then need to be encoded again, which is one reason it beats SMOTE on efficiency.

Why HSL is a better fit for federated robotics

The paper presents HSL as a more practical dynamic-encoder mechanism for edge robotics because it avoids the heavier backpropagation-style procedure used by TrainableHD while still adapting the projection matrix to local client data.

Results

The paper evaluates on actual robotic tasks, not only synthetic federated benchmarks.

The experiments run on four Wheeltec Mecanum robots powered by Raspberry Pi 4 boards. That is important context: the reported gains are coming from a genuinely constrained hardware setup, not from server-class training infrastructure.

The paper is strongest where robotics and HD structure intersect

HVO works well because hypervectors already encode a holistic view of the input. Sampling in hyperspace is not just a convenience trick; it is aligned with how HDC represents data and therefore less noisy than naive raw-feature interpolation.

The privacy claim is architectural rather than benchmarked by attack success

The paper argues that dynamic encoders reduce the reconstruction risks associated with fixed projection matrices. That is a meaningful systems argument even though the evaluation here focuses more heavily on learning performance and efficiency than on direct attack measurements.

The real deployment value is edge-side practicality

What makes this page distinct from a generic federated-learning paper is the end device: Raspberry Pi-class mobile robots with LiDAR. The contribution is therefore as much about lightweight deployment as it is about federated optimization itself.

Takeaways

Why this work fits the broader CELL direction.

  1. Point 01
    This is a clean example of CELL using hyperdimensional computing where its structural properties actually matter: low-cost representation, robust manipulation in hyperspace, and simple aggregation across clients.
  2. Point 02
    The project is also one of the lab's clearest embedded-AI stories. The target is not a datacenter model but a fleet of small robots that need collaborative learning without centralized raw data collection.
  3. Point 03
    For the website, this page should read as an algorithm-plus-deployment paper. HSL and HVO are the technical moves, but the real significance is that they make federated robot learning feasible on constrained hardware.
