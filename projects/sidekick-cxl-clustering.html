DAC 2023

Sidekick treats CXL disaggregation as a policy problem, not just a bigger memory pool.

The paper builds a type-2 CXL near-data-processing system for clustering and then adds the missing systems layer: an automated migration engine that learns which functions, threads, and memory allocations should run near the remote memory. Instead of handcrafting one accelerator per clustering algorithm, Sidekick searches the offloading space with program-context analysis and a genetic algorithm.

Abstract

The real contribution is not only that clustering likes NDP. It is that Sidekick finds the right NDP work automatically across very different clustering codes.

CXL memory disaggregation helps clustering workloads overcome local-memory capacity limits, but it also introduces remote-memory latency that can erase the benefit of the larger memory pool. Sidekick answers that tension with a type-2 CXL design where a lightweight processor near the remote memory executes the parts of clustering that would otherwise generate expensive remote accesses.

The key move is software, not only hardware. Sidekick profiles a clustering program through program contexts, tracks both computation and memory-allocation sites, and uses a genetic algorithm to decide which work should migrate to the CXL-side near-data processor. That lets the same framework support many clustering algorithms in Scikit-learn without needing source-code annotations or a handcrafted offload policy for each one.

Motivation

Three systems problems make clustering on disaggregated memory harder than “just attach more RAM.”

The paper is strong because it does not stop at saying that clustering is memory-bound. It explains why CXL alone is not enough and why automation is necessary.

Remote CXL memory fixes capacity but worsens latency

Clustering repeatedly revisits large data structures, for example during pairwise distance computation or neighborhood traversal. CXL lets the host see larger remote memory, but every remote access becomes more expensive, so naive disaggregation can simply trade one bottleneck for another.

Different clustering algorithms need different offload points

K-Means, DBSCAN, OPTICS, spectral clustering, and agglomerative methods do not stress the system in the same way. A fixed hand-tuned NDP strategy does not generalize well across algorithms, and source-code-dependent optimization is too brittle for real software stacks.

Useful migration decisions span both compute and memory contexts

The decision is not only which function to run near memory. It also includes which thread contexts and which allocation sites should live on the CXL device. The search space is large enough that an automated optimizer becomes part of the architecture.

Pipeline

Sidekick works because it learns an offloading policy from program behavior rather than assuming one bottleneck fits every clustering algorithm.

The page should make the policy loop explicit. This is not only a CXL hardware paper. It is a profiling, search, and runtime-application paper.

Step 01
Expose remote memory through a type-2 CXL NDP device
The host sees a large remote DRAM pool on the CXL device while a lightweight general-purpose processor near that memory can execute selected clustering work. The hardware intent is to reduce data motion by moving memory-intensive stages closer to where the data already live.
Step 02
Profile the workload through program contexts
Instead of relying on source annotations, Sidekick identifies functions, thread spawns, and memory-allocation sites by hashing their call-stack context. The same malloc or function call can mean different things depending on how it is reached, so the call chain becomes the right granularity for migration decisions.
Step 03
Encode migration scenarios as chromosomes
Computation contexts and memory contexts are represented as binary vectors that indicate whether each item stays on the host or migrates to the CXL device. Sidekick bootstraps these vectors by inheriting likely memory placement from the parent computation context, which reduces the search burden of a fully random start.
Step 04
Search for the best policy with a genetic algorithm
Each candidate migration policy is evaluated on system-level metrics such as IPC, execution time, or remote-request count. Crossover and mutation generate improved candidates over multiple iterations until the policy converges on a migration pattern suited to the target clustering algorithm and optimization goal.
Step 05
Apply the learned policy to large-scale runs
The policy is learned using a relatively small synthesized dataset, then applied to much larger Medium, Large, and Extra Large workloads. At runtime, Sidekick migrates the chosen computation and allocation tasks, hiding much of the remote-memory latency that plain CXL disaggregation would otherwise expose.

Why program context is the key abstraction

Sidekick is not simply tracing hot functions. It is distinguishing why a function or allocation is being used at a specific point in the call tree. That is what lets the same framework support multiple clustering algorithms without manual instrumentation.

Why the GA matters beyond “automatic tuning”

The search space spans both computation migration and allocation migration, and those decisions interact. The GA is what lets Sidekick optimize for different goals, such as IPC or remote-request reduction, instead of freezing one baked-in heuristic.

Results

The results show that the right migration policy matters more than having CXL alone.

That is the central message: memory disaggregation without NDP still leaves too many remote accesses on the table. Sidekick's value is in turning disaggregated memory into an adaptive execution substrate.

What the paper shows about policy quality and memory behavior

Question Reported evidence Interpretation
Does Sidekick beat plain disaggregated memory? Yes. The paper reports up to 92% execution-time improvement and major remote-request reductions across the evaluated clustering algorithms. CXL capacity alone is not enough. The system needs policy-driven offload to hide the remote-memory tax.
Does the policy learned on small data generalize? Yes. A policy derived from a 16 MB synthesized dataset still improves Medium, Large, and Extra Large runs with average dataset sizes of 4 GB, 8 GB, and 20 GB. The profiling phase is lightweight enough to be practical while still capturing stable workload structure.
How good is the GA policy compared to a human-crafted one? In the KMeans case study, the GA-derived Sidekick policy reduces remote CXL requests more effectively than the handcrafted migration policy and clearly beats the naive initial policy. The automation is not only convenient; it is competitive with manual tuning even on a well-understood algorithm.
Where does the speedup come from? The execution-time breakdown shows that remote-request reduction accounts for most of the benefit, contributing about 74% of the average improvement and reaching 92% for Agglomerative clustering. Sidekick wins by cutting remote traffic, not by claiming an unrealistic compute-only acceleration story.

The paper is really about workload portability across clustering algorithms

Many NDP proposals are persuasive for one kernel and then brittle beyond it. Sidekick's main achievement is that the same profiling and GA machinery can steer KMeans, DBSCAN, OPTICS, spectral clustering, and agglomerative variants without a separate hand-built accelerator strategy for each.

Remote-request reduction is the most honest systems metric here

IPC and execution time matter, but the remote-request count explains why the improvements happen. The paper is convincing because it keeps tying performance back to the actual CXL bottleneck: too many expensive remote-memory interactions.

Sidekick reframes CXL as a programmable memory-compute island

The type-2 CXL device is not treated as passive expansion memory. It becomes a place where selected program contexts can run, allocate memory, and relieve the host of the worst traffic patterns. That is the architectural idea the project page should foreground.

Takeaways

Why this page matters in CELL's systems thread.

  1. Point 01
    Sidekick is one of the clearest examples on the site of systems-software co-design. The hardware provides the CXL NDP substrate, but the real value appears only after the migration policy is learned and applied.
  2. Point 02
    For the website, this should read as a memory-systems page, not a generic clustering page. The focus is on disaggregation, remote-access avoidance, and policy automation.
  3. Point 03
    Chronologically, it also broadens the project list beyond HD learning by showing how CELL applies the same efficiency mindset to large-memory systems and data-intensive analytics.
